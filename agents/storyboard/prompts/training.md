# [2단계] 스토리보드 생성 에이전트 학습 파일
## 영화 샷 문법 + 화면 구성 원칙 통합 레퍼런스

> **이 문서의 용도**: 스토리보드 생성 에이전트의 시스템 프롬프트에 주입하거나 참조 문서로 사용한다.
> 씬 대본 → 샷 리스트 분해(사이즈/앵글/무빙/길이/설명) → 콘티 이미지 생성(PART 7) → 팀 표준 워드 파일 출력(PART 8)까지의 전 단계에서 이 문서의 원칙을 따른다.
> 이 문서는 『필름메이커의 눈』(구스타보 메르카도) 원문의 복제가 아니라, 업계 공통 촬영 문법을 실무용으로 재정리한 것이다. 각 샷의 실제 영화 사례 연구가 필요하면 해당 도서를 직접 참고할 것.

---

# PART 1. 샷 사이즈 (Shot Size) — 무엇을 얼마나 보여줄 것인가

샷 사이즈는 "정보량"과 "감정 밀도"의 트레이드오프다. 넓을수록 정보(공간·관계)가 많고, 좁을수록 감정이 짙어진다.

| 코드 | 명칭 | 프레임 기준 | 주된 기능 | 심리 효과 |
|---|---|---|---|---|
| EWS/WS | 익스트림 와이드/와이드 (Extreme Wide / Wide) | 인물이 풍경 속 점~작은 크기 | 공간·세계관 소개, 씬 설정(establishing) | 고립감, 왜소함, 장엄함 |
| FS | 풀 샷 (Full Shot) | 머리부터 발끝까지 | 신체 전체의 동작, 인물과 공간의 관계 | 객관적 관찰 |
| MS | 미디엄 샷 (Medium Shot) | 허리 위 | 대화의 기본 단위, 표정+제스처 동시 전달 | 사회적 거리(일상 대화 거리) |
| MCU | 미디엄 클로즈업 (Medium Close-Up) | 가슴 위 | 대화에서 감정 비중을 높일 때 | 친밀감 시작 |
| CU | 클로즈업 (Close-Up) | 얼굴 전체 | 감정의 절정, 중요한 반응, 결심의 순간 | 강한 친밀감·압박감 |
| ECU | 익스트림 클로즈업 (Extreme Close-Up) | 눈·입·손 등 일부 | 결정적 디테일 강조, 극도의 긴장 | 불안, 강박, 집착 |
| INS | 인서트 (Insert) | 사물·소품 클로즈업 | 정보 전달(편지, 시계, 무기), 복선 심기 | 관객만 아는 정보 → 서스펜스 |

## 사이즈 선택 규칙
1. **씬의 감정 곡선을 따라간다**: 감정이 고조될수록 점점 타이트하게(WS→MS→CU). 절정 직전의 CU가 가장 힘이 세다.
2. **새 공간은 넓게 연다**: 관객이 처음 보는 장소는 WS/EWS로 위치 관계를 먼저 심는다(establishing shot). 단, 의도적으로 공간을 숨겨 불안감을 만들 때는 예외.
3. **CU는 아껴 쓴다**: 클로즈업을 남발하면 절정에서 쓸 카드가 없어진다. 한 씬에서 감정의 최고점 1~2곳에만.
4. **ECU는 이유가 있어야 한다**: 그 디테일이 플롯이나 심리에 결정적일 때만. 장식용 ECU는 리듬만 깨뜨린다.

# PART 2. 카메라 앵글 (Angle) — 어디서 바라볼 것인가

앵글은 피사체에 대한 "관객의 심리적 위치"를 정한다.

| 코드 | 명칭 | 정의 | 심리 효과 | 대표 용도 |
|---|---|---|---|---|
| eye-level | 아이레벨 | 피사체 눈높이 | 중립, 동등한 관계 | 일상 대화, 기본값 |
| low | 로우 앵글 | 아래에서 위로 | 피사체가 강하고 위압적으로 보임 | 권력자·위협의 등장, 승리의 순간 |
| high | 하이 앵글 | 위에서 아래로 | 피사체가 약하고 왜소해 보임 | 패배, 무력감, 감시당하는 느낌 |
| dutch | 더치 앵글 | 수평선을 기울임 | 불안정, 혼란, 무언가 잘못됨 | 심리적 붕괴, 긴장, 광기 |
| overhead | 오버헤드/버즈아이 | 수직 부감 | 신적 시점, 운명론적 거리감, 패턴화 | 죽음·의식(儀式) 장면, 지형 전달 |
| POV | 시점 샷 | 인물의 눈으로 봄 | 관객=인물 동일시 | 발견의 순간, 공포(보이지 않는 것) |
| OTS | 오버 더 숄더 | 상대의 어깨 너머 | 대화의 공간적 연결 | 투샷 대화의 기본 문법 |

## 앵글 선택 규칙
1. **기본값은 아이레벨**: 특별한 의도가 없으면 아이레벨. 앵글의 힘은 "평소와 다를 때" 나온다.
2. **권력 관계를 시각화한다**: 두 인물의 힘의 차이는 로우(강자)/하이(약자) 교차로 표현. 힘의 역전이 일어나는 씬이라면 앵글도 씬 중간에 뒤집는다.
3. **더치는 극약**: 한 씬에 한 번, 심리적으로 정말 무너지는 순간에만. 남발하면 뮤직비디오가 된다.
4. **POV 앞뒤에는 보는 사람의 얼굴**: POV 샷은 "누가 본다(CU) → 무엇을 본다(POV) → 반응(CU)" 3샷 세트로 써야 관객이 시점을 놓치지 않는다.

# PART 3. 카메라 무빙 (Movement) — 어떻게 움직일 것인가

움직임은 에너지다. 정지는 관찰이고, 이동은 감정이입이다.

| 코드 | 명칭 | 정의 | 심리 효과 | 대표 용도 |
|---|---|---|---|---|
| static | 스태틱 (고정) | 카메라 고정 | 안정, 객관, 관찰자 시점 | 대화, 코미디(프레임 안에서 벌어지는 일), 정적 긴장 |
| pan | 팬 | 좌우 회전 | 시선의 이동, 공간 연결 | 인물 따라가기, 공간 훑기, 반전 드러내기 |
| tilt | 틸트 | 상하 회전 | 위계·규모의 드러남 | 건물·인물의 위용 소개(아래→위), 몰락(위→아래) |
| dolly-in | 달리 인 | 피사체를 향해 전진 | 감정으로 빨려 들어감, 깨달음 | 결심·충격의 순간 (줌인보다 몰입감 강함) |
| dolly-out | 달리 아웃 | 피사체에서 후퇴 | 고립, 버려짐, 상황의 전모 공개 | 절망의 순간, 씬의 마무리 |
| tracking | 트래킹/트러킹 | 피사체와 나란히 이동 | 동행감, 에너지 | 걷는 대화, 추격, 롱테이크 |
| handheld | 핸드헬드 | 손으로 들고 촬영(흔들림) | 불안, 즉흥성, 다큐적 현장감 | 혼란·전투·심리 불안 씬 |
| crane/jib | 크레인 | 수직 대이동 | 장엄함, 신의 시점으로 확장/하강 | 오프닝·엔딩, 스케일 과시 |
| zoom | 줌 | 렌즈 화각 변화 | 인공적 주목(공간감 왜곡) | 급박한 발견, 복고적 연출 |
| arc | 아크 | 피사체 주위를 원형 이동 | 세계가 인물을 중심으로 돎, 고조 | 각성·재회의 순간 |

## 무빙 선택 규칙
1. **움직임에는 동기가 필요하다**: 카메라가 움직이면 관객은 "왜?"를 느낀다. 인물의 이동, 감정의 변화, 정보의 공개 중 하나가 동기가 되어야 한다. 동기 없는 무빙은 멀미다.
2. **달리 인은 내면, 달리 아웃은 상황**: 들어가면 인물의 마음속으로, 빠지면 인물이 처한 상황의 전모로.
3. **핸드헬드는 씬 단위로**: 샷 하나만 핸드헬드면 실수처럼 보인다. 쓰려면 씬 전체의 질감으로.
4. **정지의 힘**: 계속 움직이다가 멈추는 순간이 가장 큰 방점이 된다. 무빙 씬의 클라이맥스는 오히려 스태틱일 수 있다.

# PART 4. 화면 구성 (Composition) — 프레임 안을 어떻게 채울 것인가

샷 설명(description)을 쓸 때 반영해야 할 구도 원칙.

1. **3분할 법칙 (Rule of Thirds)**: 프레임을 가로세로 3분할한 교차점에 주요 피사체를 배치. 인물의 눈은 상단 1/3 선에.
2. **헤드룸 (Headroom)**: 머리 위 여백. 너무 많으면 인물이 가라앉고, 없으면 답답하다. 타이트한 샷일수록 헤드룸을 줄인다.
3. **리드룸/노즈룸 (Lead Room)**: 인물의 시선·이동 방향 앞쪽에 여백을 준다. 여백을 의도적으로 반대에 주면 "막힌 느낌"(심리적 압박)을 만든다.
4. **180도 법칙**: 두 인물의 대화에서 카메라는 두 사람을 잇는 가상선(액션 라인)의 한쪽에만 머문다. 넘으면 좌우가 뒤집혀 관객이 혼란. 의도적으로 넘을 때는 관계의 붕괴를 의미하는 연출.
5. **프레임 속 프레임**: 문틀·창문·거울로 인물을 한 번 더 가두면 감금·고립·관음의 뉘앙스.
6. **전경/중경/후경 레이어**: 세 층이 살아있는 프레임이 깊이가 있다. 전경 요소(문틀, 어깨, 사물)를 하나 넣는 것만으로 입체감이 생긴다.
7. **균형과 불균형**: 대칭 구도는 질서·통제·경직을, 비대칭은 자연스러움·불안을 전달한다.
8. **시선의 무게**: 관객의 눈은 밝은 곳, 얼굴, 움직임, 프레임 중앙 순으로 끌린다. 중요한 정보를 이 위계에 맞춰 배치한다.

# PART 5. 씬 → 샷 분해 문법 (스토리보드의 핵심 로직)

## 5-1. 커버리지 기본 패턴 (대화 씬)
표준 대화 씬의 최소 커버리지:
1. **마스터 샷** (WS/FS) — 씬 전체를 담는 기준 샷. 공간과 배치를 심는다
2. **투샷** (MS) — 두 인물을 한 프레임에
3. **OTS A** — A의 어깨 너머 B
4. **OTS B** — B의 어깨 너머 A
5. **싱글 CU A / CU B** — 감정의 절정 순간용

**리듬 규칙**: 마스터로 열고 → 대화가 깊어질수록 OTS → 싱글 CU로 조여 들어간다. 갈등이 풀리면 다시 넓힌다.

## 5-2. 씬 유형별 샷 분해 가이드

| 씬 유형 | 권장 샷 구성 | 핵심 원칙 |
|---|---|---|
| 공간 소개 | EWS/WS → 디테일 INS 2~3개 → 인물 진입 FS | 넓은 것에서 좁은 것으로. 공간의 성격을 말해주는 디테일 선정 |
| 대화 (2인) | 마스터 → 투샷 → OTS 교차 → 감정 절정에 CU | 180도 법칙 유지. 감정 곡선 따라 타이트하게 |
| 대화 (3인 이상) | 마스터 → 그룹샷 → 발화자 싱글 + 리액션 샷 | 리액션 샷이 대화의 진짜 정보. 누가 어떻게 듣는가 |
| 액션/추격 | 넓은 지형샷 1 → 짧은 이동샷 다수(tracking/handheld) → 충돌 순간 CU/ECU | 지형을 먼저 심어야 액션이 읽힌다. 샷 길이는 짧게 |
| 서스펜스 | 정보 격차 설계: 관객만 아는 INS → 모르는 인물의 MS → 다가가는 POV | 히치콕 원칙: 폭탄을 보여주고 인물은 모르게 하라 |
| 감정 붕괴 | 고정 MS 길게 → dolly-in 또는 CU 전환 → 마지막 dolly-out | 붕괴 순간은 오래 버틴다. 빠져나오며 고립을 보여준다 |
| 발견/반전 | 인물 CU → POV(발견 대상) → 리액션 CU (+ dolly-in) | 3샷 세트. 반전의 크기만큼 리액션에 시간을 준다 |
| 몽타주/경과 | 통일된 규칙(같은 앵글·같은 무빙)의 짧은 샷 반복 | 반복의 규칙성이 시간 경과를 만든다 |

## 5-3. 샷 개수와 길이 배분
- **씬당 샷 수**: 대화 씬 4~8개, 액션 씬 8~15개가 기본 범위. 감정 씬은 적고 길게, 액션은 많고 짧게.
- **샷 길이(duration)**: 정보가 많은 샷(WS)은 길게(4~8초), 감정 샷(CU)은 내용에 따라(리액션 2~3초, 절정 5초+), 액션 샷은 짧게(1~3초).
- **리듬 설계**: 샷 길이의 변화 자체가 리듬이다. 균일한 길이는 단조롭고, 점점 짧아지는 배열은 긴장 고조를 만든다.

## 5-4. 연결 원칙 (샷과 샷 사이)
1. **30도 법칙**: 같은 피사체를 잇는 두 샷은 앵글이 30도 이상 달라야 점프컷처럼 보이지 않는다. 사이즈나 앵글 중 하나는 확실히 바꾼다.
2. **시선 매칭 (Eyeline Match)**: 인물이 프레임 밖을 보면, 다음 샷은 그 시선이 향한 대상이어야 한다.
3. **동작 연결 (Match on Action)**: 동작 중간에 컷하면 이음새가 숨는다.
4. **좌우 방향 유지**: 이동하는 인물의 화면상 진행 방향(좌→우 등)은 씬 내에서 유지한다. 뒤집으면 "돌아가는" 것으로 읽힌다.

# PART 6. 에이전트 프롬프트 적용 규칙

## 6-1. 샷 리스트 생성 시
```
[시스템 프롬프트에 포함할 것]
1. 씬 유형을 먼저 판별하고 (대화/액션/서스펜스/감정/발견/공간소개/몽타주)
   PART 5-2의 해당 패턴을 기본 뼈대로 사용할 것
2. 씬의 감정 곡선을 파악해 사이즈 변화(넓게→좁게 또는 그 역)에 반영할 것
3. 각 샷의 선택 이유가 PART 1~3의 심리 효과와 일치할 것
   (예: 패배 순간 → high angle, 결심 순간 → dolly-in + CU)
4. CU/ECU/dutch는 절정 순간에만 배치하고 남발하지 말 것
5. description에는 구도 요소(전경/배치/시선 방향)를 한 가지 이상 포함할 것
6. 연속 샷 간 30도 법칙·시선 매칭·방향 유지를 지킬 것
```

## 6-2. 샷 필드 값 규격
- size: EWS/WS/FS/MS/MCU/CU/ECU/INS 중 하나
- angle: eye-level/low/high/dutch/overhead/POV/OTS 중 하나
- movement: static/pan/tilt/dolly-in/dolly-out/tracking/handheld/crane/zoom/arc 중 하나
- duration: 초 단위 정수 (PART 5-3 배분 기준)
- description: 프레임 안에 보이는 것 + 구도 1요소 + (필요시) 무빙의 동기
- **dialogue**: 이 컷 동안 발화되는 대사 원문 (6-2-1 배정 규칙 참조. 없으면 빈 문자열)
- **audio**: 대사 외 음향 — 효과음, 앰비언스, 음악 지시 (씬 대본의 청각 정보에서 추출)
- **sceneSlug**: 소속 씬의 슬러그라인 (다운스트림 타이틀 카드·워드 씬 구분용)

### 6-2-1. 컷별 대사 배정 규칙 (애니매틱 파이프라인 필수)
씬 대본의 대사를 샷에 배정하는 기준 — 하류의 애니매틱 에이전트가 대사 길이로 컷 duration을 산정하므로 반드시 지킨다:
1. **발화 컷 우선**: 대사는 기본적으로 말하는 인물이 프레임에 있는 컷에 배정한다.
2. **리액션 컷 배정**: 연출 의도가 "듣는 사람의 반응"이면 대사를 리액션 컷에 배정할 수 있다 (화면 밖 음성). 이 경우 description에 "(O.S.) ~의 대사 위로"를 명시.
3. **긴 대사의 분할**: 한 대사가 여러 컷에 걸치면 대사를 문장 단위로 잘라 각 컷에 나눠 배정한다. 임의 축약 금지 — 대사 원문을 보존해야 발화 시간 계산이 정확하다.
4. **비어 있는 컷**: 대사가 없는 컷의 dialogue는 빈 문자열. audio(효과음)와 혼동하지 않는다.

## 6-3. 품질 자가 점검 체크리스트 (생성 후 검증용)
- [ ] 씬의 첫 샷이 공간/상황을 심는가? (또는 숨기는 의도가 명확한가)
- [ ] 사이즈 흐름이 감정 곡선과 일치하는가?
- [ ] CU가 씬당 1~2회 이내로 절제되어 있는가?
- [ ] 모든 무빙에 동기가 있는가?
- [ ] 연속 샷 간 30도 법칙이 지켜지는가?
- [ ] 대화 씬에서 180도 법칙이 유지되는가?
- [ ] 샷 길이 배분에 리듬 변화가 있는가?

# PART 7. 스토리보드 이미지 생성 파이프라인 (Claude + Gemini 나노바나나)

샷 리스트(텍스트 명세)를 실제 콘티 이미지로 변환하는 단계의 규칙.

## 7-1. 역할 분담 (중요)

| 역할 | 담당 | 이유 |
|---|---|---|
| 샷 리스트 생성 (size/angle/movement/description) | Claude API | 언어·구조 이해가 핵심인 작업 |
| 샷 명세 → 이미지 생성 프롬프트 변환 | Claude API | 촬영 문법을 이미지 프롬프트 언어로 번역 |
| 실제 이미지 생성 | Gemini API (나노바나나, gemini-2.5-flash-image 계열) | **Claude/Anthropic API는 이미지 생성을 지원하지 않음** |

파이프라인: `씬 대본 → [Claude] 샷 리스트(JSON) → [Claude] 샷별 이미지 프롬프트 → [Gemini] 콘티 이미지 생성 → 사람 검토`

## 7-2. 샷 명세 → 이미지 프롬프트 변환 규칙 (Claude 담당)

샷 리스트의 각 필드를 이미지 프롬프트 요소로 1:1 번역한다.

| 샷 필드 | 프롬프트 번역 예 |
|---|---|
| size: WS | "wide shot, full environment visible, subject small in frame" |
| size: CU | "close-up on face, shallow depth of field" |
| angle: low | "low angle looking up at subject, imposing" |
| angle: overhead | "top-down overhead view" |
| movement | 정지 이미지이므로 무빙 자체는 번역 불가 → 화살표/모션블러 지시로 대체: "motion blur suggesting forward movement" 또는 콘티 관례상 프레임에 화살표 주석 |
| description | 프레임 안 내용을 그대로: 인물 위치, 시선 방향, 전경 요소 |

**프롬프트 표준 템플릿**:
```
[스타일 지정] + [샷 사이즈] + [앵글] + [피사체와 행동] + [배경/전경 요소] + [조명/시간대] + [비율]
예: "black and white storyboard sketch, rough pencil style, medium close-up,
low angle, a woman in her 20s clutching an old script, dusty theater stage
in background, single overhead stage light, 16:9"
```

## 7-3. 콘티 이미지의 스타일 원칙
1. **러프가 정답이다**: 완성 일러스트가 아니라 흑백 스케치/펜슬 스타일로 통일한다. 콘티의 목적은 구도·배치 전달이지 아트워크가 아니다. 스타일 문구를 모든 프롬프트에 동일하게 고정("black and white storyboard sketch, rough pencil style")하면 씬 간 톤이 통일된다.
2. **비율 고정**: 최종 영상 비율(16:9 등)을 모든 샷에 동일하게 지정한다.
3. **한 프레임 = 한 정보**: 샷 하나의 이미지에는 그 샷의 핵심 하나만. 여러 순간을 욱여넣지 않는다.

## 7-4. 캐릭터 일관성 전략 (나노바나나 활용)
씬이 바뀌어도 같은 인물로 보이게 하는 것이 콘티 이미지의 최대 난제다.
1. **캐릭터 레퍼런스 시트 선행 생성**: 본 콘티 생성 전에 캐릭터별 기준 이미지(정면/측면, 특징 명시)를 먼저 1장 생성해 확정한다.
2. **레퍼런스 이미지 입력**: 나노바나나는 입력 이미지를 참조한 생성·편집을 지원하므로, 매 샷 생성 시 캐릭터 레퍼런스 이미지를 함께 입력해 동일 인물을 유지한다.
3. **텍스트 앵커 병행**: 프롬프트에도 캐릭터 고정 문구를 동일하게 반복한다(예: "the same woman: short black hair, gray coat, 20s"). 이미지 참조 + 텍스트 앵커를 함께 쓰면 일관성이 가장 높다.
4. **씬 단위 배치 생성**: 같은 씬의 샷들은 연속으로 생성하고, 첫 샷 결과가 좋으면 그 이미지를 다음 샷의 참조로 연쇄 입력하는 방식도 유효하다.

## 7-5. Gemini API 호출 규격 (구현 참고)
```javascript
// 예시: Gemini 이미지 생성 호출 (Google AI Studio API 키 필요)
const response = await fetch(
  "https://generativelanguage.googleapis.com/v1beta/models/" +
  "gemini-2.5-flash-image:generateContent?key=" + GEMINI_API_KEY,
  {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      contents: [{
        parts: [
          { text: imagePrompt },           // 7-2에서 변환된 프롬프트
          // 캐릭터 일관성을 위한 참조 이미지 (선택):
          // { inline_data: { mime_type: "image/png", data: base64RefImage } }
        ]
      }]
    })
  }
);
// 응답의 inline_data(base64)를 디코딩해 이미지로 표시
```
- 모델명은 배포 시점에 변경될 수 있으므로 구현 시 Google 공식 문서에서 최신 이미지 생성 모델명을 확인할 것.
- API 키는 클라이언트 코드에 노출하지 말고 서버/프록시를 경유시키는 것이 원칙. (프로토타입 단계에서 사용자 입력으로 받는 경우, 키가 브라우저에 남지 않게 세션 메모리에만 보관)

## 7-6. 에이전트 적용 규칙 (이미지 생성 단계)
```
[시스템 프롬프트에 포함할 것 — 프롬프트 변환 시]
1. 샷 필드를 7-2 표에 따라 번역하고, 표준 템플릿 순서를 지킬 것
2. 스타일 문구와 비율은 프로젝트 전체에서 고정된 값을 그대로 사용할 것
3. 캐릭터 등장 샷에는 캐릭터 앵커 문구를 반드시 포함할 것
4. movement는 정지 이미지로 번역 불가 → 모션 표현 또는 생략을 명시적으로 결정할 것
5. 생성 실패/부적합 이미지는 프롬프트를 수정해 재시도하되 3회 초과 시 수동 검토로 전환 (기존 예외 처리 정책과 동일)
```

## 7-7. 품질 체크리스트 (이미지 생성 후)
- [ ] 샷 리스트의 size/angle이 이미지에 실제로 반영되었는가?
- [ ] 같은 캐릭터가 씬 전체에서 동일 인물로 보이는가?
- [ ] 스타일·비율이 전체 콘티에서 통일되어 있는가?
- [ ] description의 핵심 정보(시선 방향, 전경 요소)가 프레임에 있는가?

# PART 8. 워드 파일 출력 규격 (팀 표준 스토리보드 .docx)

**원칙: 스토리보드 1건(에피소드/씬 묶음) = 워드 파일 1개.** 아래 규격은 팀에서 실제 사용 중인 스토리보드 워드 파일을 분석해 정의한 것이다.

## 8-1. 문서 규격
- 용지: A4 세로, 여백 1인치(상하좌우)
- 문서 최상단: 프로젝트 전역 연출 메모 1줄 (예: 캐릭터별 기본 앵글 규칙 — "방진은 기본적으로 로우 앵글, 이순신은 하이 앵글")
- 본문: 5컬럼 표. 페이지가 넘어가면 표를 나눠 새 표로 시작하되 헤더 행 반복

## 8-2. 표 구조 (5컬럼)

| 컬럼 | 폭 비율 (DXA 기준) | 내용 |
|---|---|---|
| Cut | 690 | 컷 번호 (씬 전체 연속 번호) |
| Video | 2925 | 콘티 이미지 (기본 폭 ≈1.89인치, 16:9 근사 비율) |
| Content | 3030 | 1행: 샷 표기 / 2행: 행동·연출 설명 (한국어) |
| Audio | 1530 | 대사·효과음·음악 (예: "새소리 /바람에 흔들리는 나뭇잎 소리") |
| Time | 840 | 예상 길이 (대부분 공란, duration 값이 있으면 초 단위 기입) |

## 8-3. 팀 고유 샷 표기 관례 (Content 칸 1행)
- 기본 형식: `사이즈[, 앵글][, 무빙]` — 예: `FS, Top Angle`, `MCU, Tilt Up`, `MS, Low Angle`
- **팀 고유 표기**: `OSS`(= 본 문서의 OTS, 오버 더 숄더), `TS`(투샷), `MCS`(미디엄 클로즈 샷)
- **샷 내 전환 표기**: `CU → FS` 처럼 화살표로 한 컷 안에서의 사이즈 변화를 표기 (zoom/dolly 결과를 축약)
- 에이전트는 JSON의 size/angle/movement를 이 관례로 변환해 기입한다:
  - `OTS` → `OSS`로 표기
  - `overhead` → `Top Angle`로 표기
  - movement는 `Zoom in`, `Dolly in`, `Pan Right`, `Tilt Up` 식의 관례 표기 사용

## 8-4. 이미지 배치 규칙
- **기본 컷**: Video 칸 안에 폭 ≈1.89인치(16:9) 이미지 1장
- **강조 컷 (이미지가 칸을 넘어가는 경우)**: 해당 행에서 Video+Content 칸을 가로 병합(gridSpan)하고 폭 3.9~5인치의 대형 이미지 배치. 샷 표기·설명은 병합칸 내 이미지 아래 또는 Audio 칸으로 이동
- **강조 컷 선정 기준**: 씬의 감정 절정 컷, 복잡한 구도의 액션 컷, 새 공간을 여는 설정 샷 — 씬당 0~2개 권장
- 한 컷에 프레임 진행을 보여줄 필요가 있으면(무빙 표현) Video 칸에 이미지 2장을 세로로 배치 가능

## 8-5. 샷 리스트 JSON → 워드 필드 매핑

| JSON 필드 | 워드 위치 | 변환 규칙 |
|---|---|---|
| shotNumber | Cut 칸 | 씬 경계와 무관하게 문서 전체 연속 번호로 재부여 |
| (생성 이미지) | Video 칸 | 나노바나나 생성 이미지, 16:9, 기본 폭 1.89in |
| size + angle + movement | Content 1행 | 8-3 관례 표기로 변환 (예: {size:"FS", angle:"overhead"} → "FS, Top Angle") |
| description | Content 2행 | 행동·연출 설명 그대로 |
| (대본의 대사/음향) | Audio 칸 | 씬 대본에서 해당 컷 구간의 대사·효과음 추출 |
| duration | Time 칸 | 값이 있으면 "N초", 없으면 공란 |

## 8-6. 파일 생성 규칙
- 파일명: `{프로젝트명}_스토리보드_{씬범위 또는 에피소드}.docx`
- 씬이 바뀔 때: 표를 끊지 않고 이어가되, 씬 구분 행(슬러그라인)을 표 위에 삽입하거나 새 표로 시작
- 문서 상단 연출 메모: 프로젝트 설정(캐릭터별 기본 앵글 등)이 있으면 자동 포함
